# 运行时错误分类与界面映射

## 1. 文档目的

本文档对项目全部错误码进行分类，明确每个错误码的触发条件、严重等级、界面展示位置和用户可执行的恢复动作。

本文档覆盖三个错误域:
- 运行时初始化与运行错误(RuntimeErrorCode,17 码)
- 回放错误(ReplayErrorCode,4 码)
- Patch 协议错误(PatchErrorCode,11 码)

Step 40.4 起,所有错误统一通过 `errorStore` + `ErrorToast` + `ErrorBoundary` + `app.config.errorHandler` 四层体系展示(详见 §6.4)。

---

## 2. 错误域总览

| 错误域 | 类型定义位置 | 触发阶段 | 结构化载体 | 工厂/分类函数 |
|--------|------------|---------|-----------|--------------|
| 运行时初始化与运行 | `runtime/types.ts` → `RuntimeErrorCode` | `initRuntime()` / 渲染 / 编译 / 导出 / 持久化 | `RuntimeErrorInfo` | `createRuntimeError()` / `classifyError()` |
| 回放 | `runtime/types.ts` → `ReplayErrorCode` | `replayFrame()` | `ReplayErrorInfo` | `createReplayError()` |
| Patch 协议 | `compiler/ir/patch.ts` → `PatchErrorCode` | `applyPatch()` | `PatchError` | `new PatchError()` |

错误工厂与分类逻辑统一位于 `shared/errors.ts`:
- `createRuntimeError(code, message)` — 自动填充 severity / source / recoverable / timestamp
- `classifyError(caughtError, defaultSource?)` — 将未知错误归一化为 `RuntimeErrorInfo`
- `matchErrorCode(message)` — 基于关键词的消息→错误码匹配

## 3. 运行时错误(RuntimeErrorCode)

### 3.1 错误码清单(17 码)

| 错误码 | 触发条件 | 严重等级 | 可恢复 | 来源 |
|--------|---------|---------|--------|------|
| `runtime/webgpu-unavailable` | `navigator.gpu` 不存在 | fatal | 否 | device-init |
| `runtime/adapter-unavailable` | `requestAdapter()` 返回 null | fatal | 否 | device-init |
| `runtime/context-unavailable` | `canvas.getContext('webgpu')` 返回 null | fatal | 否 | device-init |
| `runtime/device-request-failed` | `adapter.requestDevice()` 抛异常 | fatal | 是 | device-init |
| `runtime/output-texture-creation-failed` | `createOutputTexture()` 抛异常 | fatal | 是 | device-init |
| `runtime/present-pipeline-creation-failed` | `createPresentPipeline()` 抛异常 | fatal | 是 | device-init |
| `runtime/shader-compilation-failed` | 着色器编译失败 | error | 是 | render |
| `runtime/pipeline-creation-failed` | 管线创建失败 | error | 是 | render |
| `runtime/buffer-creation-failed` | 缓冲区创建失败 | error | 是 | render |
| `runtime/texture-creation-failed` | 纹理创建失败 | error | 是 | render |
| `runtime/dispatch-failed` | GPU 调度失败 | error | 是 | render |
| `runtime/gpu-device-lost` | GPU 设备丢失 | fatal | 否 | render |
| `runtime/compile-error` | RenderIR 编译错误(无可见图层 / 未知 opcode 等) | error | 是 | compile |
| `runtime/export-failed` | 渲染导出失败 | error | 是 | export |
| `runtime/persistence-failed` | IndexedDB 持久化失败 | warning | 是 | persistence |
| `runtime/unknown` | 未分类的异常 | error | 否 | unknown |

> 注:`ErrorSeverity` = 'fatal' | 'error' | 'warning' | 'info';`ErrorSource` = 'device-init' | 'compile' | 'render' | 'replay' | 'export' | 'persistence' | 'patch' | 'unknown'。

### 3.2 界面展示映射

运行时错误通过两条路径展示:

**路径 A(传统路径,阶段二建立):**

| 展示位置 | 字段 | 展示内容 |
|---------|------|---------|
| CanvasWorkspace 底部错误栏 | `store.error` | 错误消息字符串 |
| InspectorPanel → 运行时状态 → 状态 | `store.status` | 显示 `error` |
| InspectorPanel → 运行时状态 → 回放错误 | `store.replayError` | 仅在回放时触发时显示 |
| Store 内部结构化字段 | `store.runtimeError` | `{ code, message, severity, source, recoverable, timestamp }` |

**路径 B(Step 40.4 统一通知层):**

| 展示位置 | 数据来源 | 展示内容 |
|---------|---------|---------|
| ErrorToast 右上角通知 | `errorStore.activeError` | 错误码 + 消息 + 来源 + 可恢复 + userHint,可展开详情 |
| ErrorBoundary fallback | `onErrorCaptured` 捕获 | 组件级渲染错误 fallback(⚠ 图标 + message + 重试按钮) |
| 浏览器控制台 | `console.error('[App Error]', err)` | 完整错误堆栈(调试用) |

## 4. 回放错误(ReplayErrorCode)

### 4.1 错误码清单(4 码)

| 错误码 | 触发条件 | 严重等级 | 可恢复 |
|--------|---------|---------|--------|
| `replay/runtime-unavailable` | `runtime.value === null` 时调用 `replayFrame` | fatal | 是 |
| `replay/missing-data` | 帧记录不存在,或缺少 `artifact` / `compileContextSnapshot` | warning | 否 |
| `replay/incompatible-artifact-version` | `artifactSchemaVersion` 与当前版本不一致 | warning | 否 |
| `replay/signature-mismatch` | 回放时渲染签名与历史帧记录的签名不一致 | error | 是 |

### 4.2 界面展示映射

| 展示位置 | 字段 | 展示内容 |
|---------|------|---------|
| InspectorPanel → 运行时状态 → 回放状态 | `store.replayStatus` | `idle` / `success` / `error` |
| InspectorPanel → 运行时状态 → 回放错误 | `store.replayError` | 错误消息字符串 |
| Store 内部结构化字段 | `store.replayErrorInfo` | `{ code, message, severity, source, recoverable, timestamp }` |
| 帧数据面板 → 一致性校验 | `payload.renderVerificationSnapshot` | `一致` / `不一致` / `未校验` |
| ErrorToast 右上角通知(Step 40.4) | `errorStore.activeError` | 回放错误也会推送到统一通知层 |

### 4.3 用户恢复动作建议

| 错误码 | 用户可执行动作 |
|--------|--------------|
| `replay/runtime-unavailable` | 等待运行时初始化完成后再回放 |
| `replay/missing-data` | 该帧数据不完整,无法回放,尝试选择其他帧 |
| `replay/incompatible-artifact-version` | 该帧由旧版本生成,需重新渲染当前帧 |
| `replay/signature-mismatch` | 回放结果与历史记录不一致,可能运行环境已变化 |

## 5. Patch 协议错误(PatchErrorCode)

### 5.1 错误码清单(11 码)

| 错误码 | 触发条件 | 严重等级 |
|--------|---------|---------|
| `IR_PATCH_VIOLATION` | 通用 patch 结构违规 | 可恢复:patch 被拒绝 |
| `IR_STATIC_BOUNDARY_VIOLATION` | 违反静态边界硬约束(time/frame/animated 等禁用字段) | 可恢复:patch 被拒绝 |
| `IR_PATCH_TARGET_NOT_FOUND` | targetId 在 IR 中不存在 | 可恢复:patch 被拒绝 |
| `IR_PATCH_DUPLICATE_ID` | add 时 id 已存在 | 可恢复:patch 被拒绝 |
| `IR_PATCH_DANGLING_REF` | remove 时仍有引用 | 可恢复:patch 被拒绝 |
| `IR_PATCH_SCHEMA_MISMATCH` | params 不匹配 opcode schema | 可恢复:patch 被拒绝 |
| `IR_PATCH_PATH_NOT_ALLOWED` | paramKey 不在白名单 | 可恢复:patch 被拒绝 |
| `IR_PATCH_INVALID_VALUE` | 数值越界(radius < 0 / NaN 等) | 可恢复:patch 被拒绝 |
| `IR_PATCH_ATOMIC_INCOMPLETE` | AtomicTopologyPatch 缺字段 | 可恢复:patch 被拒绝 |
| `IR_PATCH_BATCH_NESTED` | PatchBatch 嵌套 PatchBatch | 可恢复:patch 被拒绝 |
| `IR_PATCH_TRANSACTION_CONFLICT` | 原子 patch 与普通 patch 冲突 | 可恢复:patch 被拒绝 |

### 5.2 界面展示映射

Patch 错误当前通过以下路径传播(Step 40.4 后也会经 errorStore 展示):

| 展示位置 | 字段 | 展示内容 |
|---------|------|---------|
| CanvasWorkspace 底部错误栏 | `store.error` | 错误消息字符串(applyDemoPatch 失败时写入) |
| 帧记录 | `record.error` | 错误消息写入帧记录的 error 字段 |
| 帧记录 | `record.status` | 帧状态标记为 `error` |
| ErrorToast 右上角通知(Step 40.4) | `errorStore.activeError` | 被 catch 的 PatchError 经 classifyError 归一化后推送 |

### 5.3 用户恢复动作建议

所有 Patch 错误统一建议:检查 patch 参数合法性,修正后重新提交。

## 6. 错误码与界面展示映射策略

### 6.1 展示原则

1. **致命错误优先**:运行时初始化错误(fatal)优先于回放错误和 patch 错误展示
2. **结构化优先**:store 内部同时保存结构化字段(`runtimeError` / `replayErrorInfo`)和字符串字段(`error` / `replayError`),界面层优先消费字符串,诊断层优先消费结构化
3. **不覆盖原则**:回放错误不覆盖运行时初始化错误;运行时初始化错误存在时,回放操作直接返回 `replay/runtime-unavailable`
4. **帧记录完整**:无论成功还是失败,每次渲染和回放都写入帧记录,失败帧记录 `status = 'error'` 并填充 `error` 字段

### 6.2 界面展示分工(传统路径)

| 界面区域 | 负责展示的错误类型 | 数据来源 |
|---------|------------------|---------|
| CanvasWorkspace 状态指示器 | 运行时状态(idle/initializing/ready/error) | `store.status` |
| CanvasWorkspace 底部错误栏 | 当前运行时错误或 patch 错误 | `store.error` |
| InspectorPanel → 运行时状态 | 运行时状态、画布尺寸、输出格式、回放状态、回放错误 | `store.status` / `store.replayStatus` / `store.replayError` |
| InspectorPanel → 补丁调试 | 当前场景、图层、指令、补丁编号、补丁摘要 | `store.currentScenario` / `store.currentLayerId` 等 |
| InspectorPanel → 设备能力 | 存储格式、纹理尺寸限制、计算限制 | `store.capability` |
| 帧数据面板 → 帧摘要 | 帧编号、时间戳、耗时、状态、补丁摘要、工件版本 | `FrameSnapshot` |
| 帧数据面板 → 一致性校验 | 渲染签名状态、签名说明 | `FrameSnapshot.renderVerificationState` |

### 6.3 错误码到用户消息映射

运行时错误和回放错误的消息已在 `device.ts` 和 `runtime.ts` 中硬编码,与错误码一一对应。界面层直接展示 `message` 字段,不做二次翻译。

Patch 错误的消息在 `patch.ts` 的 `PatchError` 构造函数中生成,格式为 `Patch validation failed: {首个违规描述}`。

`shared/errors.ts` 提供 `getErrorCodeLabel(code)` 与 `getSeverityLabel(severity)` 用于人类可读名称转换。

### 6.4 Step 40.4 统一通知层(errorStore + ErrorToast + ErrorBoundary)

Step 40.4 引入四层错误处理体系,统一所有未捕获错误的展示:

| 层级 | 组件/文件 | 职责 | 触发时机 |
|------|----------|------|---------|
| L1 全局兜底 | `main.ts` → `app.config.errorHandler` | 捕获所有未处理的 Vue 组件错误 | 任何未 catch 的组件错误 |
| L2 组件边界 | `ErrorBoundary.vue` → `onErrorCaptured` | 捕获子组件错误,展示 fallback UI | 子组件渲染/生命周期错误 |
| L3 通知队列 | `errorStore.ts` → Pinia Store | 错误归一化 + LRU 队列(50 上限)+ activeError | push / pushStructured / pushMessage |
| L4 用户通知 | `ErrorToast.vue` | 右上角 Toast 展示 activeError | errorStore.activeError 变化时 |

**错误归一化流程:**
```
未知错误(throw/console.error/window.alert)
  → errorStore.push(caught, userHint?)
  → classifyError(caught, defaultSource?)  [shared/errors.ts]
  → RuntimeErrorInfo { code, severity, source, recoverable, timestamp }
  → severityToLevel(severity)  [fatal/error → 'error', warning → 'warning', info → 'info']
  → ErrorEntry { id, code, message, level, source, recoverable, timestamp, acknowledged, userHint }
  → errorStore.queue(LRU 50)+ errorStore.activeError
  → ErrorToast.vue 展示
```

**NotificationLevel 三级样式:**
| 级别 | 颜色 | 自动消失 | 图标 |
|------|------|---------|------|
| error | 红 #2a1517 | 否(需手动关闭) | ✕ |
| warning | 黄 #2a2515 | 是(5s) | ! |
| info | 蓝 #15202a | 是(5s) | i |

**App.vue 已替换的 window.alert 调用点:**
| 调用点 | 原 behavior | 新 behavior |
|--------|------------|------------|
| handleOpenProject catch | `window.alert(e)` | `errorStore.push(e, '请检查项目文件是否损坏')` |
| handleDropProject 校验失败 | `window.alert(formatValidationResult)` | `errorStore.pushMessage(formatValidationResult, 'error')` |
| handleDropProject 校验警告 | `console.warn(msg)` | `errorStore.pushMessage(msg, 'warning')` |
| handleDropProject catch | `window.alert(e)` | `errorStore.push(e, '拖拽导入失败,请检查文件格式')` |

## 7. 错误码版本管理

### 7.1 新增错误码

新增错误码时必须:
1. 在类型定义中追加(`RuntimeErrorCode` / `ReplayErrorCode` / `PatchErrorCode`)
2. 在 `shared/errors.ts` 的映射表中补充(`RUNTIME_ERROR_SEVERITY` / `RUNTIME_ERROR_RECOVERABLE` / `RUNTIME_ERROR_SOURCE`)
3. 在本文档对应章节补充清单
4. 在触发代码中用 `createRuntimeError()` 或 `new PatchError()` 包装
5. 补充对应测试用例

### 7.2 废弃错误码

废弃错误码时必须:
1. 保留类型定义(不删除,标注 `@deprecated`)
2. 在本文档标注废弃原因和替代码
3. 确保已有测试仍通过

### 7.3 当前版本

- RuntimeErrorCode: 17 个码(含 `runtime/unknown` 兜底)
- ReplayErrorCode: 4 个码
- PatchErrorCode: 11 个码
- errorStore 队列上限: 50(LRU 淘汰)
