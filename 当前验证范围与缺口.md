# 当前验证范围与缺口

## 1. 当前验证目标

当前项目的验证目标是：

验证从 Render IR 到 WebGPU 呈现，再到帧记录、历史帧选择、历史帧回放与前端消费的整条主链路是否可行。

这不是最终产品验收，而是工程链路验收。

---

## 2. 当前已经验证的范围

### 2.1 编译与 Patch 协议层

已验证内容：

- Render IR 基础结构
- 静态边界约束
- Patch 协议合法性校验
- PatchEngine 应用行为
- batch 回滚语义
- 错误码边界清晰性

对应测试：
- `src/compiler/ir/__tests__/patch.test.ts`
- `src/compiler/ir/__tests__/patchEngine.test.ts`

当前状态：
- 已通过自动化测试

---

### 2.2 WebGPU 初始化与渲染主链路

已验证内容：

- WebGPU 是否可初始化
- 画布上下文是否可创建
- 输出纹理是否可创建
- 区域渲染工件是否可提交给求值器
- 求值结果是否可通过 present pass 显示到画布

当前状态：
- 已具备代码路径
- 已在构建层面通过
- 仍缺少自动化验证与更细的错误分类

---

### 2.3 帧记录与前端消费层

已验证内容：

- 每次渲染后记录 RuntimeFrameRecord
- 将记录适配为 FrameSnapshot
- 时间轴可消费帧列表
- 数据面板可展示当前帧摘要
- 当前帧可导出 JSON
- 帧适配层自动化测试已补齐
- 播放控制器自动化测试已补齐

当前状态：
- 已具备前端数据链路
- 当前更多是结构验证，不是最终交互形态

---

### 2.4 历史帧回放入口层

已验证内容：

- `runtimeStore.replayFrame(frame)` 端口已存在
- 选中时间轴帧时已接入回放调用路径
- store 已区分 `selectedFrame` 与 `presentedFrame`
- 帧记录已补回放相关字段
- 回放状态字段 `replayStatus` 与 `replayError` 已建立
- 工件版本校验已进入回放逻辑

当前状态：
- 代码路径已建立
- store 回放自动化测试已补齐

---

### 2.5 区域工件编译层

已验证内容：

- 线性渐变工件生成
- 噪声图层默认值回退
- 不可见图层跳过逻辑
- 不支持指令报错
- 无可见图层报错

当前状态：
- 自动化测试已补齐

---

## 3. 当前没有完全验证的部分

### 3.1 历史帧真实回放闭环

当前虽然已经可以：
- 选中某一帧
- 调用 `replayFrame(frame)`
- 用历史工件尝试重新渲染
- 更新当前展示帧编号
- 更新回放状态与回放错误

但还没有完全证明：
- 画布一定稳定回到该帧对应画面
- 检查面板所有字段都严格切到历史帧语义
- 回放字段在更复杂场景下足够稳定

这是当前最核心的未闭环点。

---

### 3.2 帧数据是否足够长期支持回放

当前帧记录中已经补了：
- `artifact`
- `renderIrSnapshot`
- `compileContextSnapshot`
- `artifactSchemaVersion`

但尚未证明这些数据是否足够长期支持：
- 任意帧回放
- 连续播放回放
- 局部编辑后重新生成历史片段
- 后续工件结构升级后的兼容回放

因此仍需继续确认：
- 快照方案是否最终冻结
- 是否仍需引入 patch 链作为备用方案

---

### 3.3 运行时错误体系

当前 patch 层已有较成熟的错误码与文档。
当前回放层已经有：
- `replayStatus`
- `replayError`
- 工件版本不兼容分支

但 runtime 层整体仍主要依赖字符串错误。

尚未完成的内容：
- 初始化错误分层
- 实时渲染错误分类
- 回放错误分类进一步结构化
- 导出错误分类

---

### 3.4 自动化测试覆盖的最后缺口

当前自动化测试已经覆盖：

1. patch 协议层
2. 帧适配层
3. 播放控制层
4. store 回放层
5. 区域工件编译层

尚缺少：

1. WebGPU 相关最小行为模拟测试
2. 更完整的运行时初始化失败测试
3. 画布历史态与检查面板历史态的一致性验证

这意味着“渲染 -> 记录 -> 回放 -> 时间轴”主链路已经有自动保护，但还没有覆盖所有运行时边缘情况。

---

## 4. 当前最值得补的内容

按优先级建议如下：

### 优先级 1：补运行时初始化失败与错误分类测试

现在比继续堆 UI 更值钱的是把 runtime 错误路径补齐。

### 优先级 2：固定回放最小字段集合

必须明确以下字段是否就是冻结版最小集合：
- `artifact`
- `renderIrSnapshot`
- `compileContextSnapshot`
- `artifactSchemaVersion`

### 优先级 3：补历史态一致性验证

优先补：
- 选中历史帧后画布回放行为的更细粒度验证
- 检查面板是否完整投影到历史态

### 优先级 4：补运行时错误规范

建议先沉淀文档，再决定是否做代码枚举化。

---

## 5. 当前阶段不建议优先做的事情

下面这些事情当前都不是最高优先级：

- 进一步打磨视觉界面
- 继续扩大时间轴交互细节
- 增加更多展示型模块
- 做完整产品化文案

原因很简单：
目前最重要的是证明链路闭环，而不是界面完整度。

---

## 6. 当前结论

项目当前已经验证了：
- Patch 协议层
- 最小渲染链路
- 帧记录到界面消费链路
- 帧适配与播放控制链路
- 历史帧回放代码入口
- store 回放路径
- 区域工件编译规则

项目当前仍未完全验证：
- 历史帧真实回放闭环
- 回放所需最小字段集合是否最终稳定
- 运行时错误体系
- WebGPU 相关边缘路径

因此，当前最合理的工作方向是继续补“运行时闭环能力”和“历史态一致性验证”，而不是继续扩展界面层。
