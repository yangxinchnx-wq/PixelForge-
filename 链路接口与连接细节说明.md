# 链路接口与连接细节说明

## 1. 文档目的

本文档直接对应当前代码实现，定义已经存在的接口、连接点、调用顺序、输入输出结构、已打通链路和未收口断点。

目标是让后续开发者可以直接根据本文档定位：

- 哪个模块向哪个模块提供数据
- 数据结构长什么样
- 当前链路具体在哪里接起来
- 哪些连接已经打通
- 哪些连接已经有入口但还没测试收口

---

## 2. 主链路总图

当前主链路按顺序如下：

1. 页面挂载，拿到画布 DOM
2. `runtimeStore.initialize(canvas)` 初始化运行时
3. `initRuntime()` 创建 GPU 上下文、输出纹理和呈现管线
4. `renderCurrentIR()` 生成编译上下文
5. `compileRenderIRToRegionArtifact(currentIr)` 输出区域渲染工件
6. `createRegionEvaluator(...).render(artifact)` 把工件写入 GPU 缓冲区并执行计算
7. `renderFrame()` 将输出纹理呈现到画布
8. `appendFrameRecord()` 将本次结果记录为 `RuntimeFrameRecord`
9. `InMemoryFrameRepository` 保存帧记录
10. `createFrameSnapshotList()` 将帧记录转换为 `FrameSnapshot`
11. `toTimelineFrame()` 和 `createFrameSections()` 把快照提供给时间轴和数据面板
12. `useFramePlaybackController()` 基于帧快照提供播放、暂停、步进能力
13. `runtimeStore.replayFrame(frame)` 在选中历史帧时尝试回放
14. `replayStatus / replayError` 记录回放结果
15. `renderVerificationSnapshot` 用于验证历史帧回放一致性

## 3. 模块接口定义

## 3.1 页面入口层

### 文件
- `src/App.vue`

### 职责
- 创建画布引用
- 调用 `runtimeStore.initialize()`
- 将 `runtimeStore` 的状态传给各界面模块
- 将帧仓储内容适配为时间轴和数据面板输入
- 在时间轴选帧时同时触发选中和回放
- 通过 `createInspectorSections()` 生成检查面板正式投影

### 当前仍未收口的点
- 页面层仍然显式负责“选中 + 回放”双调用，后续可进一步收口到 store 或控制器
- 检查面板已抽离正式投影函数，但仍可继续细化字段投影策略

- `applyDemoPatch(color)`
- `renderCurrentIR(runtimeOverride?)`

### 当前实现要点
- 已通过 `createRuntimeStore(frameRepository)` 支持仓储注入
- 默认导出 `useRuntimeStore = createRuntimeStore(new InMemoryFrameRepository())`
- 回放失败与实时渲染失败已分层：
  - `error` 用于通用运行时错误
  - `replayStatus / replayError` 用于回放状态与回放错误
- 回放时会进行三层校验：
  - 工件版本校验
  - 回放字段完整性校验
  - 渲染签名一致性校验

### 关键内部连接

#### 连接 A：初始化连接
`initialize()` -> `initRuntime()` -> `renderCurrentIR()`

#### 连接 B：补丁连接
`applyDemoPatch()` -> `applyPatch()` -> `currentIr = outcome.ir` -> `renderCurrentIR()`

#### 连接 C：帧记录连接
`renderCurrentIR()` -> `appendFrameRecord()` -> `frameRepository.upsertFrame(record)`

#### 连接 D：选帧连接
`selectFrame(frame)` -> `selectedFrame.value = frameRepository.getFrame(frame)?.frame ?? null`

#### 连接 E：回放连接
`replayFrame(frame)` -> `frameRepository.getFrame(frame)` -> `executeArtifactRender(...)` -> `presentedFrame.value = frame`

#### 连接 F：回放状态连接
- 缺少回放字段 -> `replayStatus = 'error'`、`replayError = '当前帧缺少回放所需数据'`
- 工件版本不兼容 -> `replayStatus = 'error'`、`replayError = '工件版本不兼容: ...'`
- 渲染签名不一致 -> `replayStatus = 'error'`、`replayError = '历史帧回放结果与记录签名不一致'`
- 回放成功 -> `replayStatus = 'success'`、`replayError = null`

### 当前问题
- 回放状态已经显式化，但还没有细分错误类别枚举
- 仍未进入像素级输出一致性验证，仅停留在渲染签名级别

## 3.8 帧记录端口

### 文件
- `src/runtime/types.ts`
- `src/stores/runtime.ts`

### 帧记录结构

#### `RuntimeFrameRecord`
```ts
{
  frame: number
  timestampMs: number
  durationMs: number
  status: 'idle' | 'initializing' | 'ready' | 'error'
  scenario: string
  layerId: string | null
  opcode: string | null
  patchId: string | null
  patchSummary: string | null
  canvasSize: RuntimeCanvasSize | null
  outputFormat: string | null
  error: string | null
  artifact?: RegionCompileArtifact
  renderIrSnapshot?: RenderIR
  compileContextSnapshot?: CompileContextSnapshot
  artifactSchemaVersion?: string
  payload?: {
    compileHints?: Record<string, JsonLiteral>
    capabilitySummary?: Record<string, string>
    renderVerificationSnapshot?: {
      descriptorData: number[]
      auxData: number[]
      canvasWidth: number
      canvasHeight: number
      seed: number
      time: number
    }
      [key: string]: unknown
  }
}
```

### 当前写入时机
- 渲染成功时，已经写入：
  - `artifact`
  - `renderIrSnapshot`
  - `compileContextSnapshot`
  - `artifactSchemaVersion`
  - `payload.renderVerificationSnapshot`

### 当前问题
- 当前帧编号是前端本地计数，不是外部统一时序编号
- `timestampMs` 使用 `performance.now()`，只在当前会话内有意义
- 若后续要做会话持久化或跨端同步，需要补 `sessionId`、`sourceId`、`recordedAt` 等字段

## 3.11 播放控制端口

### 文件
- `src/services/frame/playback.ts`

### 当前行为细节
- 若没有选中帧，播放前会自动选第一帧并触发回放
- 到达最后一帧时自动停止
- 若外部选中的帧已经不存在，会自动回退到第一帧并触发回放
- 播放与步进都会同时调用 `onSelectFrame()` 和 `onReplayFrame()`
- 播放控制器测试已补齐

### 当前问题
- 当前它已经不是纯播放控制器，而是“播放 + 回放触发器”
- 但回放成功与否不在控制器层判断，仍依赖 store 和 runtime

## 6. 当前结论

当前项目已经完成的是：
- 最新态渲染链路
- 最新态帧记录链路
- 最新态时间轴消费链路
- 历史帧回放入口
- 历史工件重渲染尝试
- store 回放测试注入
- 回放状态显式化
- 工件版本校验入口
- 检查面板历史态正式投影
- 渲染签名一致性验证

当前项目还没有完全完成的是：
- 更细粒度的运行时错误分类
- 像素级画布结果一致性验证
- 更复杂 WebGPU 边缘路径测试

因此，后续所有开发都应该围绕一个核心问题展开：

“当前帧记录结构，是否足以稳定重建任意历史帧的真实渲染结果，并且能被自动化测试持续证明？”
